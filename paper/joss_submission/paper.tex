\section{Summary}\label{summary}

\texttt{HydraR} is a lightweight, state-managed framework designed to
orchestrate complex ``agentic'' workflows---autonomous, multi-step
processes driven by Large Language Models (LLMs) natively within R.
While mainstream orchestration tools such as LangChain (Chase 2022) and
CrewAI (Moura 2023) are predominantly Python-based, \texttt{HydraR}
fulfills the R community's critical requirements for rigorous
reproducibility, auditability, and seamless integration with established
statistical pipelines.

Researchers can use \texttt{HydraR} to design workflows as step-by-step pipelines (Directed Acyclic Graphs) or looping state machines, where each step can be an AI-driven task, a predictable R function, or an automated data auditor. \texttt{HydraR} ensures broad compatibility---connecting seamlessly to cloud AI engines (Gemini (Google 2025), Claude, OpenAI) and local models (Ollama)---while enforcing reliability through a built-in validation engine that rigorously inspects the configuration for syntax and structural flaws prior to running.

\section{Statement of Need}\label{statement-of-need}

The contemporary R ecosystem lacks a unified framework that provides both high-level coordination for AI agents and the safe management of local files. While existing packages, such as \texttt{ellmer} (Wickham et al. 2025), focus primarily on conversational interfaces and \texttt{mall} (Couch 2024) excels at applying LLM operations to structured data, neither is equipped to securely save the workflow's progress (its ``state'') or provide the isolated workspaces needed for long-running, multi-agent collaborations.

\texttt{HydraR} addresses three fundamental challenges in this domain:
1. \textbf{Auditability}: It maintains a central memory system that continuously saves workflow checkpoints (using the DuckDB database (Raasveldt and Mühleisen 2019)), ensuring a complete, verifiable record of all AI interactions and data changes. 2. \textbf{Filesystem Safety}: By leveraging Git worktrees, \texttt{HydraR} creates temporary, isolated project branches for each agent working in parallel. This prevents file corruption and race conditions when multiple tasks run simultaneously. 3. \textbf{Portable Workflows}: By defining agent logic using standard \texttt{YAML} files and \texttt{Mermaid.js} diagrams, \texttt{HydraR} separates the workflow's design from the code that runs it. This text-based blueprint acts like an ``OpenAPI for Agents''---a shareable format that is easy to audit in R and can be exported to other AI development frameworks like LangGraph or Google's Agent Development Toolkit.

\section{State of the Field}\label{state-of-the-field}

While tools like \texttt{reticulate} (Ushey et al. 2024) allow researchers to connect R with popular Python-based AI frameworks (such as LangChain), this approach often creates compatibility issues when translating complex statistical data structures between the two programming languages. \texttt{HydraR} avoids this translation barrier by being built entirely using R's native architectural frameworks (Chang 2021). This native design ensures that scientific data remains intact and allows researchers to use their familiar error-checking tools. Furthermore, existing AI tools in R generally serve different purposes: \texttt{ellmer} (Wickham et al. 2025) focuses on interactive chat features rather than automated workflows, while \texttt{gptstudio} (Nivard and Wade 2024) functions as an interactive coding assistant. In contrast, \texttt{HydraR} is specifically designed to manage long-running, autonomous research tasks where multiple AI agents work in parallel and securely save their progress.

\section{Software Design}\label{software-design}

\texttt{HydraR} employs a highly modular \texttt{R6} architecture
spanning several key components: - \textbf{Node implementation details}: The \texttt{AgentLLMNode} and \texttt{AgentLogicNode} \texttt{R6} classes encapsulate LLM-driven prompts and deterministic pure-R logic steps, respectively. They utilize pluggable \texttt{AgentDriver} backends for provider-agnostic model communication.
- \textbf{Network representation and traversal engine}: The \texttt{AgentDAG} module is the core orchestrator that manages task dependencies and validates the logical structure of the workflow (via \texttt{igraph} (\citeproc{ref-igraph}{Csárdi \& Nepusz, 2006})). It coordinates both sequential and parallel operations. For instance, when multiple agents execute simultaneously, it acts as a synchronization barrier---waiting for all concurrent processes to conclude (utilizing \texttt{furrr} (\citeproc{ref-furrr}{Vaughan \& Reznik, 2022})) before initiating any dependent downstream tasks. Additionally, it features a built-in validation engine that rigorously inspects the entire workflow configuration prior to execution. This engine cross-references the network's visual layout with the defined agent instructions and enforces strict computational safety standards, ensuring that \texttt{HydraR} provides a reliable and reproducible environment for executing complex, multi-agent protocols.
- \textbf{Centralized shared memory}: The \texttt{AgentState} component acts as a shared repository that maintains a complete historical record of the workflow. It uses tailored merging rules to systematically organize and combine the outputs produced by multiple agents.
- \textbf{Isolated execution environments}: The \texttt{WorktreeManager} is an engine that creates temporary, isolated workspaces for parallel tasks, paired with a conflict resolution tool that can automatically resolve data collisions or flag them for human review.

\subsection{Multimodal Travel Itinerary
Planner}\label{multimodal-travel-itinerary-planner}

In this example, an LLM \texttt{Planner} agent generates a draft
itinerary, which is then audited by a deterministic \texttt{Validator}
against hard constraints. Successfully validated plans proceed to a
\emph{multimodal image generation} phase where \textbf{Gemini 3.1 Flash}
(Google 2025) creates visual assets based on the itinerary's locales.
Finally, a logic node renders a bespoke, CSS-styled HTML pamphlet.
Crucially, \textbf{conditional looping} between the planner and
validator enforces an iterative refinement cycle until all strict

\subsection{Parallel Sorting Algorithm
Comparison}\label{parallel-sorting-algorithm-comparison}

simultaneously implementing different sorting algorithms.

noded systematically reconciles the independent branches back to the

the aggregated algorithms across five continuous trials

\subsection{Fault-Tolerant Pipelines with DuckDB
Persistence}\label{fault-tolerant-pipelines-with-duckdb-persistence}

To mitigate transient failures inherent to long-running scientific
workflows, \texttt{HydraR} integrates advanced checkpoint-and-resume
functionality powered by DuckDB (Raasveldt and Mühleisen 2019). Whenever
a pipeline pauses mid-execution, it automatically persists its complete
\texttt{AgentState}. Upon restart, any operator-applied programmatic
interventions are seamlessly merged, actively preventing redundant
evaluations by allowing execution to jump directly back to the paused
node.

\section{Research Impact Statement}\label{research-impact-statement}

\texttt{HydraR} is actively deployed at the \textbf{Australian Proteome
Analysis Facility (APAF)} to architect and automate resilient
bioinformatics tooling. Its declarative YAML schemas empower
non-programming domain experts to routinely govern advanced LLM logic
pipelines without any R coding proficiency, thereby democratizing the
scale and reproducibility of model-assisted scientific research.

\section{AI Usage Disclosure}\label{ai-usage-disclosure}

Transparency and accountability are core tenets of the \texttt{HydraR}
paradigm. A substantial portion of the implementation was co-developed
utilizing \textbf{Antigravity}, an agentic AI coding assistant,
supervised by a rigorous ``human-in-the-loop'' review protocol.
Specifically, the original authors independently conceptualized the
underlying system architecture, while the AI generated specialized
logic, unit tests, and software documentation with every resultant
commit sequentially tested and manually verified. To ensure software reliability, the framework is continuously evaluated against an automated testing suite consisting of 556 unit tests, currently maintaining a 100\% pass rate (zero failures).
